## v0.4.0 Requirements (Shipped)

### Policy CRUD (TUI Screens)

- [x] **POLICY-01** (Phase 16): Admin can list all policies in a scrollable table with name, priority, action, and enabled state. Table is sorted by priority ascending. Inline action keys: `n` (new), `e` (edit), `d` (delete) available from the list view without navigating to a separate menu.

- [x] **POLICY-02** (Phase 14): Admin can create a new policy via a multi-field form with: name (required text), description (optional text), priority (numeric, required), action (select: ALLOW / DENY / AllowWithLog / DenyWithAlert), and one or more typed conditions added via the conditions builder (POLICY-05). Form submits via POST /admin/policies. PolicyStore cache is invalidated after successful commit.

- [x] **POLICY-03** (Phase 15): Admin can edit an existing policy by loading it from GET /admin/policies/:id and modifying any field (name, description, priority, action, enabled flag, conditions). Form submits via PUT /admin/policies/{id}. PolicyStore cache is invalidated after successful commit.

- [x] **POLICY-04** (Phase 15): Admin can delete a policy with a confirmation prompt ("Delete policy '{name}'? [y/n]"). The delete action is reachable directly from the policy list via the `d` keypress. Form submits via DELETE /admin/policies/{id}. PolicyStore cache is invalidated after successful commit.

### Conditions Builder

- [x] **POLICY-05** (Phase 13): Admin can build policy conditions using a 3-step sequential picker (no raw JSON entry at any step). Attribute → operator → typed value. Completed conditions drop into a pending list with a delete binding. Delete-and-recreate replaces in-place edit for v0.4.0. All conditions are evaluated as implicit AND; NOT/OR is deferred to v0.5.0.

### Policy Simulation

- [x] **POLICY-06** (Phase 16): Admin can simulate a policy decision by filling an EvaluateRequest form (user_sid, user_name, groups, device_trust, network_location, path, classification, action, access_context). Form submits via POST /evaluate. Response displays matched_policy_id (or "none"), decision, and reason. Error states display descriptive text in the form.

### Import / Export

- [x] **POLICY-07** (Phase 17): Admin can export the full policy set to a JSON file. GET /policies → serde_json::to_string_pretty → native save dialog with filename `policies-export-{YYYY-MM-DD}.json`.

- [x] **POLICY-08** (Phase 17): Admin can import policies from a JSON file. Native file picker → parse JSON as Vec<PolicyResponse> → GET /policies for existing IDs → compute conflict diff → confirm screen with counts → on confirm, POST non-conflicting and PUT conflicting policies. Abort on first failure with per-policy error message. Audit events emitted server-side via existing R-09 infrastructure.

## Outcome Notes

- **Route correction:** The plan assumed `GET /admin/policies` for the
  pre-flight policy fetch. UAT surfaced a 405 Method Not Allowed —
  the server only serves `GET /policies`. Fixed in commit `7dda578`.
  POST and PUT continue to use the Phase 9 legacy `/admin/policies`
  and `/admin/policies/{id}` routes.
- **No raw JSON in the TUI:** the conditions builder completely
  eliminated raw-JSON editing for policy authoring; every field is
  typed-picker or validated text input.
