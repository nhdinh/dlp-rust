## Overview

The v0.5.0 milestone upgrades the ABAC engine and admin TUI from
implicit-AND-over-typed-conditions to flat boolean composition with
expanded per-attribute operators. It also closes the delete-and-recreate
gap left by Phase 13's conditions builder.

The work splits cleanly along the engine/TUI seam established in
v0.3.0/v0.4.0: the server gains a `mode` column, a typed `mode` field
on `PolicyPayload` / `PolicyResponse`, an evaluator switch on mode,
and an expanded operator set. The TUI gains a mode picker on the
Create/Edit forms, attribute-type-aware operator pickers in the
3-step conditions builder, and in-place editing of pending conditions.
Backward compatibility for v0.4.0-authored policies (default
`mode = ALL` on read) is delivered with the wire-format change so
the server never ships in a broken-by-default state.

## Phase Details

### Phase 18: Boolean Mode Engine + Wire Format

**Goal**: Server-side delivery of flat boolean composition. The
ABAC evaluator switches on a per-policy `mode` (`ALL` / `ANY` /
`NONE`), the SQLite schema persists it with a backward-compatible
default, and the wire format carries it round-trip through
`PolicyPayload` / `PolicyResponse`. v0.4.0-authored policies loaded
from storage without a `mode` value are treated as `ALL`, preserving
exact prior semantics. No TUI change in this phase â€” the server is
mode-aware but the TUI implicitly authors `ALL`.

**Depends on**: Phase 17 (last shipped phase; import/export typed
PolicyResponse round-trip is the contract this phase extends)

**Requirements**: POLICY-12

**Success Criteria** (what must be TRUE):
  1. `policies.mode` column exists with `NOT NULL DEFAULT 'ALL'` (or
     equivalent migration via `ALTER TABLE` in `dlp-server::db::open`),
     and existing rows from v0.4.0 are populated with `'ALL'` on
     migration without manual intervention.
  2. `PolicyPayload` (POST/PUT body) and `PolicyResponse` (GET body)
     both carry a typed `mode: PolicyMode` field; serde defaults
     `mode` to `ALL` when the field is absent on POST/PUT input.
  3. `PolicyStore::evaluate` (sync hot path, unchanged signature) honors
     the policy's mode: `ALL` returns the policy when every condition
     matches, `ANY` when at least one matches, `NONE` when no condition
     matches. Cache invalidation continues to fire on every policy
     mutation.
  4. Unit tests cover the three modes against a 3-condition policy
     and prove the legacy `ALL`-default path: a `PolicyPayload` JSON
     literal with no `mode` key deserializes successfully and evaluates
     identically to a v0.4.0 policy.

### Phase 19: Boolean Mode in TUI + Import/Export

**Goal**: Surface the boolean mode in the admin TUI and prove
end-to-end round-trip. The Policy Create and Policy Edit forms gain
a mode picker (`ALL` / `ANY` / `NONE`) above the conditions list,
defaulting to `ALL` for new policies and reflecting the loaded value
on edit. Export writes the `mode` field unconditionally; import
tolerates files that omit it (defaulting to `ALL`) and includes it
on subsequent re-export. Closes POLICY-09's user-facing contract.

**Depends on**: Phase 18 (wire format + evaluator must accept and
honor the mode field before the TUI can author it meaningfully)

**Requirements**: POLICY-09

**Success Criteria** (what must be TRUE):
  1. Policy Create form shows a `Mode: [ALL] / ANY / NONE` row above
     the conditions builder; admin selects one and the choice is
     submitted as the `mode` field of the `POST /admin/policies`
     payload.
  2. Policy Edit form pre-fills the mode row from the loaded
     `PolicyResponse.mode` and submits any change via
     `PUT /admin/policies/{id}`. PolicyStore cache is invalidated on
     submit (existing pattern).
  3. Export (`policies-export-YYYY-MM-DD.json`) includes the `mode`
     field on every policy. Importing an export written by Phase 19
     restores the same boolean semantics on the destination server
     (round-trip equality on `PolicyPayload` excluding
     `version`/`updated_at`).
  4. Importing a v0.4.0-style export file (no `mode` field on any
     policy) succeeds; every imported policy reads back with
     `mode = ALL` and evaluates identically to its source.
  5. End-to-end integration test: create three policies (one per
     mode) via the admin API, evaluate the same `EvaluateRequest`
     against each, and assert the decision differs as expected per
     the boolean semantics.

### Phase 20: Operator Expansion

**Goal**: Expand the per-condition operator set from `eq`-only to
an attribute-type-aware menu. The ABAC evaluator honors the new
operators; the conditions builder's step 2 surfaces only operators
valid for the attribute chosen in step 1. Existing `eq`-only
conditions keep working unchanged. Touches both engine and TUI but
lands together because the wire format already carries an `operator`
string field on `PolicyCondition` â€” the change is enum widening and
evaluator branches, not a schema change.

**Depends on**: Phase 19 (the mode UX is the reference for "operator
picker is filtered by context"; landing operator expansion afterwards
keeps the conditions builder UX changes coherent within one phase
boundary)

**Requirements**: POLICY-11

**Success Criteria** (what must be TRUE):
  1. The conditions builder's step 2 (operator picker) shows
     attribute-type-aware operators only:
     - `Classification`: `eq`, `ne`, `gt`, `lt`
     - `MemberOf`: `eq`, `ne`, `contains`
     - `DeviceTrust` / `NetworkLocation` / `AccessContext`: `eq`, `ne`
     Selecting an attribute that is later changed correctly resets the
     operator selection if the previously chosen operator is invalid
     for the new attribute.
  2. `PolicyStore::evaluate` honors `gt`/`lt` for `Classification`
     using the canonical T1 < T2 < T3 < T4 ordering, `ne` as the
     negation of `eq` for all attribute types, and `contains` as
     substring match for `MemberOf` group strings.
  3. v0.4.0 policies (operator = `eq` everywhere) evaluate identically
     after the change â€” no regression in PolicyStore unit tests.
  4. Unit tests cover each new operator with a representative
     positive and negative case.

### Phase 21: In-Place Condition Editing

**Goal**: Close the delete-and-recreate gap left by Phase 13. From
the pending-conditions list in the conditions builder, selecting a
condition and pressing `e` re-enters the 3-step picker pre-filled
with the existing attribute, operator, and value. On save, the
condition replaces at its original list index; on cancel, the
pending list is unchanged. Delete remains available via the
existing key. TUI-only â€” no server, wire format, or evaluator
change.

**Depends on**: Phase 20 (the operator picker filters operators by
attribute; pre-filling that picker in edit mode requires the
filter logic to be in place so a previously-chosen `gt` for a
`Classification` attribute pre-fills correctly)

**Requirements**: POLICY-10

**Success Criteria** (what must be TRUE):
  1. From the pending-conditions list, selecting a row and pressing
     `e` opens the 3-step picker with attribute, operator, and value
     pre-populated from the selected condition.
  2. Saving the edited condition replaces it at the original list
     index â€” total condition count is unchanged, position order is
     preserved, and other conditions in the list are untouched.
  3. Cancelling the edit (Esc) returns to the pending list unchanged
     â€” the original condition is intact at its original index.
  4. The existing delete binding still works on the pending list
     (no regression).
  5. Editing a condition whose attribute is then changed correctly
     resets the operator selection per Phase 20's filter logic, and
     re-prompts for a value compatible with the new attribute type.
