# Milestone v0.4.0: Policy Authoring

**Status:** SHIPPED 2026-04-20
**Phases:** 13-17
**Total Plans:** 9
**Requirements Delivered:** POLICY-01 through POLICY-08 (8/8)

## Overview

The v0.4.0 milestone delivered the full admin policy-authoring workflow
in `dlp-admin-cli`. The TUI now covers the entire policy lifecycle —
list, create, edit, delete, simulate, import, and export — backed by
the admin API and the PolicyStore cache introduced in v0.3.0.

The admin no longer needs to edit raw JSON or hit the REST API
directly to curate policies: every operation is available as a typed
form with inline validation and descriptive error reporting.

## Phases

### Phase 13: Conditions Builder

**Goal:** Provide a 3-step sequential picker for building typed
`PolicyCondition` lists without any raw JSON entry.
**Requirements:** POLICY-05
**Plans:** 2 plans

Plans:

- [x] 13-01: Data model types (ConditionAttribute, CallerScreen,
      PolicyFormState, Screen::ConditionsBuilder) and dispatch handler
      with helpers and unit tests
- [x] 13-02: Render function (draw_conditions_builder modal overlay)
      and human visual verification

**Outcome:** Step 1 lists 5 attributes (Classification, MemberOf,
DeviceTrust, NetworkLocation, AccessContext). Step 2 filters operators
per attribute (the engine still only evaluates `eq`, documented).
Step 3 is a typed value picker: T1–T4 for Classification, free text
for MemberOf SIDs, a 4-option list for DeviceTrust/NetworkLocation,
and a 2-option list for AccessContext. Completed conditions drop into
a pending-conditions list below the picker with a delete binding;
delete-and-recreate replaces in-place edit for v0.4.0.

### Phase 14: Policy Create

**Goal:** Multi-field form that creates a new policy with an attached
condition list via the conditions builder.
**Requirements:** POLICY-02
**Plans:** 2 plans

Plans:

- [x] 14-01: State model (Screen::PolicyCreate, ACTION_OPTIONS,
      form_snapshot), dispatch handler (handle_policy_create,
      action_submit_policy, CallerScreen Esc fix), and unit tests
- [x] 14-02: Render function (draw_policy_create) and visual
      verification

**Outcome:** Form captures name (required), description (optional),
priority (required numeric), action (ALLOW / DENY / AllowWithLog /
DenyWithAlert) and the built condition list. Submit posts to
`POST /admin/policies`, invalidates the PolicyStore cache on success,
and surfaces server errors inline.

### Phase 15: Policy Edit + Delete

**Goal:** Load, edit, and delete existing policies from the TUI.
**Requirements:** POLICY-03, POLICY-04
**Plans:** 1 plan

Plans:

- [x] 15: Row constants, PolicyEdit state + handlers, delete
      confirmation, and extended rendering

**Outcome:** Pressing `e` on the policy list loads the full record
via `GET /admin/policies/{id}` and populates all form fields for edit.
Submit routes through `PUT /admin/policies/{id}`. Pressing `d` shows
a "Delete policy '{name}'?" prompt and fires `DELETE` on confirm.
Cache is invalidated on both paths. Edit retains the enabled flag as
a new form toggle.

### Phase 16: Policy List + Simulate

**Goal:** Scrollable policy table and standalone evaluate-request
simulation form.
**Requirements:** POLICY-01, POLICY-06
**Plans:** 2 plans

Plans:

- [x] 16-01: PolicyList polish (column widths, n-key binding, inline

**Outcome:** Policy list shows Priority / Name / Action / Enabled,

### Phase 17: Import + Export

**Goal:** Persist and restore the full policy set via JSON files with
conflict detection.
**Requirements:** POLICY-07, POLICY-08
**Plans:** 2 plans

Plans:

- [x] 17-01: Export action, ImportConfirm screen, and PolicyMenu
      entries (rfd native file dialogs)
- [x] 17-02: Typed PolicyResponse/PolicyPayload structs and import
      execution with abort-on-error + unit tests

**Outcome:** Export fetches the live policy set, pretty-prints it,
and writes to a user-chosen path (default filename
`policies-export-YYYY-MM-DD.json`). Import opens a file picker,
parses the JSON into `Vec<PolicyResponse>`, computes a conflict diff
against the current server state, and transitions to `ImportConfirm`.
On Confirm, new IDs are POSTed and existing IDs are PUT with
abort-on-first-failure; success shows `Imported N (X new, Y updated)`.
UAT caught a 405 Method Not Allowed on `GET /admin/policies` — fixed
in commit `7dda578` by routing GETs to `/policies`.

## Milestone Summary

**Key Decisions:**

- **Typed import format:** Wave 2 introduced `PolicyResponse` and
  `PolicyPayload` structs with a `From` conversion that drops
  `version` / `updated_at`. This is the authoritative schema for
  import/export JSON.
- **Skip-nav in ImportConfirm:** the 5-row list's first three rows
  are informational (header + diff counts) and non-selectable; Up/Down
  cycles only between the Confirm and Cancel buttons.
- **Abort-on-first-failure:** import stops at the first POST/PUT
  error, showing `Failed on policy '{name}': {reason}`. Prior
  successes are already persisted server-side — the admin does not
  get a rollback.
- **Native file dialogs:** `rfd = "0.14"` chosen for native Windows
  save/open dialogs; no extra permissions required.
- **Server route asymmetry:** `GET /policies` (list) is the only
  authoritative GET route — `/admin/policies` only has POST/PUT/DELETE
  (Phase 9 legacy). This was caught in UAT.
- **TOML export blocked:** `toml` crate incompatibility with
  `#[serde(tag = "attribute")]` on `PolicyCondition`. Deferred as
  POLICY-F4.

**Issues Resolved:**

- Phase 16 Esc-key bug: PolicySimulate form cleared the edit buffer
  on Esc; fixed to preserve the committed value (commit `e1afee3`).
- Phase 17 GET path bug: `GET /admin/policies` returned 405; fixed
  to use `GET /policies` (commit `7dda578`).

**Issues Deferred (to v0.5.0 or later):**

- POLICY-F1: AND/OR/NOT boolean logic between conditions
- POLICY-F2: In-place condition editing (v0.4.0 uses delete-and-recreate)
- POLICY-F3: Per-condition operator expansion (`gt`, `lt`, `ne`,
  `contains`) — the ABAC engine only evaluates `eq`
- POLICY-F4: TOML export format (serde tag incompatibility)
- POLICY-F5: Batch import endpoint `POST /admin/policies/import` to
  avoid N cache invalidations on bulk import
- POLICY-F6: Typed `Decision` action field in server (eliminate
  `_ => DENY` silent fallback)
