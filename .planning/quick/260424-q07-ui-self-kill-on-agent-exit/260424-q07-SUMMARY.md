# Phase 260424-q07 Plan 01: UI Self-Kill on Agent Exit Summary

**One-liner:** Monotonic `pipe1_ever_connected` latch in `UiState` drives `std::process::exit(0)` when the agent pipe drops after first connection, guarded by a pure `watchdog_should_exit()` helper with three unit tests.

## What Was Built

`dlp-user-ui/src/app.rs` received three targeted changes:

1. `UiState` gained a `pipe1_ever_connected: bool` field (initialised `false` in `run_with`).
2. A new pure helper `watchdog_should_exit(currently_connected, ever_connected)` encapsulates the latch logic — returns `true` only when `ever_connected` is already `true` and `currently_connected` is `false`. This keeps the logic testable without a live iced runtime.
3. The `Message::Tick` arm now reads `pipe1_connected`, calls `watchdog_should_exit`, and invokes `std::process::exit(0)` with an `info!` log line if the helper returns `true`.

## Deviations from Plan

**1. [Rule 2 - Missing critical functionality] Extracted pure helper for unit testability**

- **Found during:** Task 1
- **Issue:** The plan specified inline code in the `Tick` arm. `DlpApp::update` calls `tray::apply_pending_status()` and `muda::MenuEvent::receiver()` which require a live Windows tray/event loop — they cannot be called from `#[test]` without a full system environment.
- **Fix:** Extracted the boolean latch logic into `watchdog_should_exit(currently_connected, ever_connected)` — a pure function with no side effects — and called it from the `Tick` arm. The `Tick` arm retains the `info!` log and `std::process::exit(0)` call as specified.
- **Files modified:** `dlp-user-ui/src/app.rs`
- **Commit:** f12d139
