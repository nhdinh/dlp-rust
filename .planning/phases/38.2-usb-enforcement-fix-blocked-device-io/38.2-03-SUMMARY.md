# Phase 38.2 Plan 03: Drive-Letter Correlation Fix Summary

**Rewrote `find_drive_letter_for_instance_id` with kernel-authoritative volume-to-disk mapping, replacing the buggy heuristic that ignored `instance_id` and returned the first unassigned fixed drive letter**

## Accomplishments

- `disk_number_for_instance_id()`: Derives disk number from SetupDi instance_id via `GUID_DEVINTERFACE_DISK` enumeration + `IOCTL_STORAGE_GET_DEVICE_NUMBER`
- `query_disk_number_for_handle()`: Sends `IOCTL_STORAGE_GET_DEVICE_NUMBER` to an open disk handle
- `find_drive_letter_for_disk_number()`: Enumerates volumes via `FindFirstVolumeW`/`FindNextVolumeW`, matches disk number via `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS`, resolves drive letter via `GetVolumePathNamesForVolumeNameW`
- `find_drive_letter_for_instance_id()`: Two-step kernel-authoritative mapping replacing the buggy heuristic
- `test_find_drive_letter_signature_uses_instance_id_not_underscore`: Unit test verifying the new function signature uses `instance_id` (no underscore prefix)

## Files Created/Modified

- `dlp-common/src/disk.rs` - Rewrote `find_drive_letter_for_instance_id` with kernel-authoritative volume-to-disk mapping (506 insertions, 206 deletions)

## Decisions Made

- Used `GetVolumePathNamesForVolumeNameW` (volume GUID path) rather than `GetVolumePathNamesForVolumeMountPointW` (mount point path) for drive letter resolution from volume GUID path
- `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS` parsed manually via byte offsets to avoid struct availability issues in the `windows` crate
- `SetupDiEnumDeviceInterfaces` + `SetupDiGetDeviceInterfaceDetailW` used to open a disk handle for `IOCTL_STORAGE_GET_DEVICE_NUMBER`
- No longer uses `DRIVE_FIXED` check — the new volume-to-disk mapping naturally filters to mounted volumes

**2. [Windows crate struct availability] Manual byte parsing for VOLUME_DISK_EXTENTS**
- **Found during:** Task 1 (Struct resolution)
- **Issue:** `VOLUME_DISK_EXTENTS` struct is not directly available in all `windows` crate versions; `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS` control code was available but the struct was not
- **Fix:** Parsed the raw bytes manually using known offsets: `NumberOfDiskExtents` at offset 0, `DISK_EXTENT` array starting at offset 8 with `DiskNumber` at offset 0 within each 24-byte extent
- **Files modified:** `dlp-common/src/disk.rs`
- **Verification:** `cargo build -p dlp-common` passes, unit tests pass
