# Phase 38.2: USB Enforcement Fix - Blocked Device I/O - Context

**Gathered:** 2026-05-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 38.2 closes the live USB-03 enforcement gap surfaced on 2026-05-05: registered `Blocked`-tier USB devices log `DENY` audit events but file writes still succeed, because the `notify`-based file watcher is observe-only and cannot fail an I/O operation.

The phase delivers **two real-time, OS-enforced layers** that block writes BEFORE they reach the disk:

1. **PnP layer** — `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE` fires for every registered Blocked USB device on every plug-in path (no race, no startup gap, no missed handler).
2. **Volume DACL layer** — Deny-all DACL applied to the volume root for the Blocked tier, as a second OS-enforced fallback if any PnP path fails.

In addition, the phase fixes the drive-letter mislabel observed in the same audit log (`detection/disk.rs` reports SanDisk USBSTOR at "C" when the user plugged at F:), since correct drive-letter resolution is required for both DACL operations and audit clarity.

Out-of-scope items are explicitly captured in `<deferred>` below — the most notable is the Windows minifilter driver, which is deferred to v0.8.0+.

</domain>

<decisions>
## Implementation Decisions

### Enforcement architecture (LOCKED via Area 1)
- **D-01:** Phase 38.2 ships **two real-time, OS-enforced layers**. Both must be live before the phase is verified. No kernel driver, no DLL injection, no signing infrastructure beyond what already exists.
- **D-02:** **Primary layer = PnP.** `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE` (already implemented in `dlp-agent/src/device_controller.rs:101`). Real-time, OS-enforced at the lowest layer (device tree). Persists across re-plugs of the same VID/PID/serial until `CM_Enable_DevNode` runs.
- **D-03:** **Secondary layer = Volume DACL deny-all.** Extend the existing `set_volume_readonly()` pattern in `device_controller.rs` to a deny-all variant for the Blocked tier. Reuse `restore_volume_acl()` for cleanup. Real-time, NTFS-enforced at the volume-root level — independent of the PnP layer.
- **D-04:** **`notify`-based file watcher demoted to observability ONLY.** May still emit BLOCK audit events for visibility, but is NOT relied on for enforcement. The actual enforcement happens at PnP + DACL before file I/O reaches the volume.
- **D-05:** **User-mode API hooking REJECTED with concrete rationale** — do not add to plan. Blockers: EDR conflict (Defender/CrowdStrike/SentinelOne quarantine third-party hooking as malware), PPL processes inaccessible, WoW64 split, AppInit_DLLs deprecated, trivially bypassed via cmd.exe / PowerShell, doesn't catch SYSTEM/kernel I/O, CET shadow-stack incompatibility on Win11. NOT real defense in depth.
- **D-06:** **Windows minifilter DEFERRED to v0.8.0+.** Multi-week kernel work (WDK install, kernel debugging, test → EV signing, INF/CAT installer, BSOD risk during dev, IPC channel design). Out of 38.2 scope per ROADMAP wording.

### Tier-change semantics (LOCKED via Area 3)
- **D-07:** **`enable_usb_device` fires on physical device removal only.** When admin changes tier `Blocked` → `ReadOnly`/`FullAccess` via the registry, the change takes effect on the next physical unplug & re-plug. NO new wiring in the 30s registry-cache poll path. Admin instructs user to re-plug. Simpler, no race window between PnP state and tier state.
- **D-08:** **`restore_volume_acl()` is symmetric — fires on physical device removal only.** When admin changes tier `ReadOnly` → `FullAccess`, the DACL restore happens via the existing `on_usb_device_removal` handler. Same physical-re-plug requirement as D-07. Symmetric design avoids tier-state vs DACL-state divergence.

### Operational scope (LOCKED via Area 2)
- **D-09:** **`AGENT-UNKNOWN` agent_id remediation is OUT of 38.2 scope.** `service.rs:759` reads `DLP_AGENT_ID` env var and defaults to `"AGENT-UNKNOWN"`. Per `docs/OPERATIONAL.md` lines 60-63 and 531, admins set this via `setx /M DLP_AGENT_ID "AGENT-XXX-001"` before service install. The "AGENT-UNKNOWN" in the live audit log is admin error, not a code bug. Split to **Phase 38.3 (operational hardening)** — startup warning when `DLP_AGENT_ID` unset, format validation, persist to registry.

### Drive-letter resolution (LOCKED via Area 4)
- **D-10:** **Drive-letter mislabel fix IN scope for 38.2.** The audit log shows `DISK_DISCOVERY` reporting SanDisk USBSTOR at drive_letter "C" when the user plugged at F:. The mislabel originates in `dlp-agent/src/detection/disk.rs` (Phase 33 disk-enum). Folded into 38.2 because: (a) without correct drive-letter resolution, DACL operations target the wrong volume, breaking the secondary enforcement layer; (b) it surfaced together with the enforcement gap in the same evidence; (c) it cleans up the audit-log noise this phase otherwise leaves behind.

### Researcher focus (from Area 1)
- **D-11:** **Enumerate every USB plug-in timing path** and prove enforcement covers all: plug-before-agent (device plugged before service starts), plug-during-startup-window (service starting up when plug arrives), plug-while-running (steady state), server-disconnected (registry cache stale or absent), agent-restart-while-plugged (service crash/restart with device already mounted). Each path must hit at least one of D-02 or D-03.
- **D-12:** **Investigate the WR-01 USB_DEVICE↔VOLUME notification race** flagged in `.planning/phases/31-usb-cm-blocking/31-VERIFICATION.md`. The Phase 31 wiring uses `pending_identity` reconciliation between `usb_wndproc` (USB_DEVICE arrival) and `handle_volume_event` (VOLUME arrival). If VOLUME fires before USB_DEVICE captures identity, `device_identities[F]` is never populated and the unregistered-device fallback path runs (which is what the live audit log shows — `usb_enforcer.rs:152` "known-USB-without-identity" branch).

### Claude's Discretion
None — every gray area resolved with explicit user decision. Researcher and planner have full lock-in.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`DeviceController::disable_usb_device`** (`device_controller.rs:101`) — already implements `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE`. The PnP layer (D-02) is REUSE, not new code. The plan's job is to ensure it FIRES for every plug-in timing path.
- **`set_volume_readonly` / `restore_volume_acl`** (`device_controller.rs`) — DACL machinery exists for the ReadOnly tier. The plan extends this pattern with a `set_volume_deny_all()` (or equivalent) for the Blocked tier (D-03). Cleanup uses the same `restore_volume_acl` (D-08).
- **`enable_usb_device`** (`device_controller.rs:158-230`) — already implemented; called from `on_usb_device_removal` only (D-07). No new call sites needed.
- **`pending_identity` reconciliation** (`detection/usb.rs`) — Phase 31's mechanism for bridging the USB_DEVICE↔VOLUME race. The plan must verify or harden this (D-12), not replace it.
- **`REGISTRY_CACHE`** (init in `service.rs`) — provides `lookup(vid, pid, serial) → Tier`. Reused as-is. Tier-change reactivity is intentionally NOT added (D-07/D-08).

### Established Patterns
- **Phase 31 USB CM blocking wiring** — `usb_wndproc` → `on_usb_device_arrival` (parks `pending_identity`) → `handle_volume_event` (takes `pending_identity`, calls `apply_tier_enforcement`) → `controller.disable_usb_device`. The plan extends this same chain to also call the DACL deny-all path for Blocked tier.
- **30s registry-cache poll cadence** (`service.rs:625`) — current poll loop refreshes the registry cache. **NOT extended** for tier-change re-enablement (D-07/D-08). The plan must explicitly avoid wiring `enable_usb_device` or `restore_volume_acl` into this poll path.
- **Audit emit via `EmitContext`** (`service.rs:759`) — `agent_id` resolved from `DLP_AGENT_ID` env var with `"AGENT-UNKNOWN"` fallback. Pattern unchanged (D-09 — out of scope; remediation in Phase 38.3).
- **Defense-in-depth fallback in `usb_enforcer.rs`** — for unregistered USBs, returns `Decision::DENY`. For registered (Blocked/ReadOnly/FullAccess), returns `None` (active enforcement is DeviceController's job). This contract is the design intent; D-02 + D-03 make it real for Blocked.

### Integration Points
- **Plug-in path:** `usb_wndproc` (kernel notification) → `on_usb_device_arrival` → `handle_volume_event` → `apply_tier_enforcement` → `DeviceController::disable_usb_device` (PnP, D-02) AND new `set_volume_deny_all` (DACL, D-03). Both layers must fire for Blocked tier.
- **Plug-out path:** `usb_wndproc` (removal) → `on_usb_device_removal` → `DeviceController::enable_usb_device` (D-07) AND `restore_volume_acl` (D-08). Symmetric cleanup.
- **Startup paths to cover (D-11):** plug-before-agent, plug-during-startup-window, plug-while-running, server-disconnected, agent-restart-while-plugged. Researcher must enumerate each and prove at least one of {D-02, D-03} catches it.
- **Drive-letter resolution (D-10):** `detection/disk.rs::DISK_DISCOVERY` enumeration. Researcher must locate the mislabel root cause (likely a stale enumeration that picks the first-letter-in-table or returns the system drive when the volume isn't yet mounted).

</code_context>

<specifics>
## Specific Ideas

- **Reuse over re-architecture.** Both enforcement layers (PnP + DACL) are already implemented in `device_controller.rs`. The plan is a wiring + path-coverage exercise, NOT a new component.
- **WR-01 race investigation is non-negotiable** (D-12). The live audit log evidence (`usb_enforcer.rs:152` firing) strongly suggests this race is happening in production — `device_identities[F]` is empty when the volume event fires. The researcher must identify the exact timing failure mode and the planner must wire the fix.
- **DACL deny-all variant** (D-03) is a small extension to the existing `set_volume_readonly` pattern. Pattern: `OpenVolumeRoot → SetSecurityInfo (DACL with deny-all ACE for Everyone) → log result`. Cleanup is the existing `restore_volume_acl`.
- **Drive-letter fix** (D-10) is constrained to `detection/disk.rs`. Do NOT touch `detection/usb.rs` for this — they're separate code paths.

</specifics>

<deferred>
## Deferred Ideas

- **Windows minifilter driver (FltMgr)** — true kernel-level I/O interception on top of PnP + DACL. Provides the only path to a per-process exception model and catches SYSTEM/kernel I/O. Multi-week scope: WDK install, kernel debugging, test → EV signing for production, INF/CAT installer, BSOD risk during dev, IPC channel from driver to user-mode agent. **Target: v0.8.0+.**
- **Per-process exception lists** — would allow specific processes (e.g., admin tooling, backup agents) through the enforcement layer. Requires the minifilter to be useful (DACL is process-agnostic). Out of 38.2.
- **`AGENT-UNKNOWN` operational hardening** — startup warning when `DLP_AGENT_ID` is unset, format validation, persist to registry. **Phase 38.3** (operational hardening). Splitting from 38.2 keeps this phase laser-focused on enforcement; admin operational issues belong in their own phase.
- **Tier-change reactivity (poll-driven re-enable)** — auto-detect tier changes in the 30s registry-cache poll and call `enable_usb_device` / `restore_volume_acl` immediately. Considered and rejected for 38.2 (D-07/D-08) due to race-window complexity. May be revisited if admin UX feedback shows physical-re-plug requirement is too friction-heavy.

</deferred>
