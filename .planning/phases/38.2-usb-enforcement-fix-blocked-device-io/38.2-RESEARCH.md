# Phase 38.2 Research: USB Enforcement Fix â€” Blocked Device I/O

**Date:** 2026-05-05
**Researcher:** Direct codebase analysis (agent spawning unreliable on Windows)

---

## Problem Analysis

The live bug (reported 2026-05-05) manifests as: a USB device registered with `trust_tier=blocked` logs `BLOCK`/`DENY` audit events, but file writes to the device still succeed. The root cause is two-fold:

1. **The `notify`-based file watcher is purely observational.** `InterceptionEngine` uses `notify::RecommendedWatcher` with a 500ms poll. It observes file events AFTER they occur. When `UsbEnforcer::check` returns `Decision::DENY`, `run_event_loop` only emits an audit log and a UI toast, then `continue`s. The file has already been written.

2. **The intended real enforcement (`DeviceController`) is not firing for this device.** The audit log shows the unregistered-device fallback path firing (`usb_enforcer.rs:152` "known-USB-without-identity" branch), meaning the device either (a) is not in `DeviceRegistryCache`, or (b) the arrival handler did not call `apply_tier_enforcement`.

Phase 38.2 delivers two real-time, OS-enforced layers:
- **PnP layer:** `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE` â€” device disabled at the device-tree level.
- **Volume DACL layer:** Deny-all DACL on the volume root â€” NTFS-enforced fallback.

Additionally, the drive-letter mislabel (`detection/disk.rs` reports SanDisk at "C" when plugged at F:) must be fixed because DACL operations target volumes by drive letter.

## Timing Path Analysis (D-11)

### Path 1: Plug-before-agent (device already plugged when service starts)

**Flow:** `service.rs:594` â†’ `detector.scan_existing_drives()` â†’ `usb.rs:86-97`

```rust
pub fn scan_existing_drives(&self) {
    for letter in 'A'..='Z' {
        if self.is_removable_drive(letter) {
            self.blocked_drives.write().insert(letter);
        }
    }
}
```

**Problem:** `scan_existing_drives` only adds drive letters to `blocked_drives`. It does NOT:
- Capture `DeviceIdentity` (VID/PID/serial) â€” `device_identities` remains empty.
- Call `apply_tier_enforcement` â€” no PnP disable, no DACL modification.

**Result:** The device is in `blocked_drives` but not in `device_identities`. When a write occurs, `usb_enforcer.rs:152` fires the "known-USB-without-identity" fallback â€” which returns `Decision::DENY` but cannot block the I/O (notify is observe-only).

**Gap:** No enforcement layer catches this path. The PnP disable never fires because `scan_existing_drives` doesn't know the device's VID/PID/serial, and the arrival handler (`on_usb_device_arrival`) only fires for new plug events, not pre-existing devices.

### Path 2: Plug-during-startup-window (service starting up when plug arrives)

**Flow:** `service.rs:641` sets `DEVICE_CONTROLLER` â†’ `service.rs:656` sets `DRIVE_DETECTOR` â†’ then `service.rs:780` spawns `device_watcher_task`.

**Problem:** There is a window between when the USB window is created and when `set_drive_detector` / `set_device_controller` are called. If a USB device plugs during this window:
- `usb_wndproc` receives `WM_DEVICECHANGE` but `DRIVE_DETECTOR` or `DEVICE_CONTROLLER` may be `None`.
- The notification is dropped silently (no retry).

**Result:** Same as Path 1 â€” device may end up in `blocked_drives` via a later `scan_existing_drives` or a subsequent volume event, but without identity capture or PnP enforcement.

### Path 3: Plug-while-running (steady state)

**Flow:** `usb_wndproc` â†’ `dispatch_usb_device_arrival` â†’ `on_usb_device_arrival` â†’ parks `pending_identity` â†’ `handle_volume_event` â†’ reconciles `pending_identity` â†’ `apply_tier_enforcement`

**Normal case:** If `GUID_DEVINTERFACE_USB_DEVICE` arrives before `GUID_DEVINTERFACE_VOLUME`:
1. `on_usb_device_arrival` captures identity and parks it in `pending_identity`.
2. `handle_volume_event` (on VOLUME arrival) takes `pending_identity`, stores it in `device_identities`, and calls `apply_tier_enforcement`.
3. `apply_tier_enforcement` looks up tier in `REGISTRY_CACHE` and calls `controller.disable_usb_device` (Blocked) or `controller.set_volume_readonly` (ReadOnly).

**Race case (WR-01):** If `GUID_DEVINTERFACE_VOLUME` arrives BEFORE `GUID_DEVINTERFACE_USB_DEVICE`:
1. `handle_volume_event` fires first. `pending_identity` is `None`.
2. The drive letter is added to `blocked_drives`, but `device_identities` has no entry.
3. When `GUID_DEVINTERFACE_USB_DEVICE` arrives later, `on_usb_device_arrival` parks the identity, but `handle_volume_event` has already processed the volume event â€” the identity is never reconciled with a drive letter.
4. Result: `device_identities[F]` is empty forever. The device is known as USB (in `blocked_drives`) but has no identity. `usb_enforcer.rs:152` fires on every write.

**This is the most likely cause of the live bug.**

### Path 4: Server-disconnected (registry cache stale or absent)

**Flow:** `apply_tier_enforcement` checks `REGISTRY_CACHE.get()`:

```rust
let tier = if let Some(cache) = REGISTRY_CACHE.get() {
    cache.trust_tier_for(&identity.vid, &identity.pid, &identity.serial)
} else {
    warn!("registry cache unavailable â€” applying default-deny (Blocked)");
    UsbTrustTier::Blocked
};
```

**Problem:** If `REGISTRY_CACHE` is not set (no server connection), the default is `Blocked`. This is correct for security. However, `disable_usb_device` is still called with the device's VID/PID/serial â€” if the device IS registered as `FullAccess` or `ReadOnly` but the cache is stale, it gets incorrectly blocked.

**Mitigation:** The default-deny is intentional (Zero Trust). The issue is cache freshness, not enforcement wiring. The 30s poll keeps the cache reasonably fresh.

### Path 5: Agent-restart-while-plugged (service crash/restart with device already mounted)

**Same as Path 1.** After restart, `scan_existing_drives` adds the drive to `blocked_drives` but does not capture identity or enforce. The device was previously disabled at PnP level, but `CM_Disable_DevNode` state is NOT persisted across service restarts â€” it persists across re-plugs of the same device, but if the service restarts, the PnP state is unknown to the new process.

**Critical gap:** Windows DOES persist the disable state in the registry (the device remains disabled in Device Manager), but the new agent process doesn't know to re-apply it. Actually, if `CM_Disable_DevNode` was called with `CM_DISABLE_ABSOLUTE`, the device IS disabled in the Windows device tree. A service restart doesn't change that â€” the device remains disabled until `CM_Enable_DevNode` runs or the device is physically removed and re-inserted.

Wait â€” let me re-check. `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE` writes to the device node's config flags in the registry. This state IS persisted across reboots. So if the device was disabled before the service restart, it remains disabled. The only issue is if the device was plugged AFTER the service started but before the arrival handler fired â€” which is Path 2.

**Revised Path 5 assessment:** If the device was already disabled before the restart, it stays disabled. If the device plugged while the service was down and is still plugged when the service restarts, Path 1 applies.

## WR-01 Race Investigation (D-12)

### Root Cause

The `pending_identity` mechanism in `usb.rs:341-346` assumes `GUID_DEVINTERFACE_USB_DEVICE` always arrives before `GUID_DEVINTERFACE_VOLUME`:

```rust
// In handle_volume_event (line 341):
if let Some(identity) = detector.pending_identity.lock().take() {
    detector.device_identities.write().insert(*letter, identity.clone());
    apply_tier_enforcement(*letter, &identity);
}
```

**Failure mode when VOLUME arrives first:**
1. `handle_volume_event` fires for DBT_DEVICEARRIVAL on VOLUME.
2. `pending_identity` is `None` (USB_DEVICE hasn't arrived yet).
3. The drive letter is added to `blocked_drives` via `on_drive_arrival`.
4. `device_identities` is NOT populated.
5. Later, `dispatch_usb_device_arrival` â†’ `on_usb_device_arrival` parks the identity.
6. But `handle_volume_event` already processed this volume event â€” there is NO second VOLUME event to trigger reconciliation.
7. The identity remains in `pending_identity` indefinitely (until the next USB device arrives and overwrites it, or until removal clears it).

**Evidence this is happening in production:**
- The audit log shows `usb_enforcer.rs:152` "known-USB-without-identity" firing.
- The debug analysis confirms the unregistered-device fallback path is active.
- The device IS plugged and recognized as USB (in `blocked_drives`), but has no identity.

### Fix Required

The reconciliation must handle BOTH orderings:
1. USB_DEVICE arrives first â†’ identity parked â†’ VOLUME arrives â†’ identity reconciled. (Current behavior â€” works.)
2. VOLUME arrives first â†’ drive letter added to `blocked_drives` â†’ USB_DEVICE arrives later â†’ identity must be reconciled with the ALREADY-KNOWN drive letter.

For case 2, `on_usb_device_arrival` needs to:
- Check if the device is already in `blocked_drives` (meaning VOLUME already arrived).
- If so, find the matching drive letter and populate `device_identities` + call `apply_tier_enforcement`.
- If not, park in `pending_identity` as before.

**Challenge:** `on_usb_device_arrival` does not know the drive letter at USB_DEVICE arrival time â€” the `dbcc_name` is a device path, not a drive letter. The drive letter is only known from the VOLUME event.

**Solution approach:** When USB_DEVICE arrives, scan `blocked_drives` for a drive letter that is:
- In `blocked_drives` (known USB)
- NOT in `device_identities` (no identity yet)
- Is a removable drive (just to be safe)

If exactly one such drive letter exists, assign the identity to it and call `apply_tier_enforcement`. If zero or multiple, park in `pending_identity` as before.

This is a heuristic but a safe one: if there's only one un-identity'd USB drive, it must be the one that just sent the USB_DEVICE notification.

## Drive-Letter Mislabel Root Cause (D-10)

### Root Cause

`dlp-common/src/disk.rs:772-809`:

```rust
fn find_drive_letter_for_instance_id(
    _instance_id: &str,          // UNUSED!
    _already_found: &[DiskIdentity],
) -> Option<char> {
    let drives = unsafe { GetLogicalDrives() };
    // ...
    for letter in 'A'..='Z' {
        // Check if fixed drive and path exists...
        // Returns the FIRST unassigned fixed drive letter!
        let already_taken = _already_found
            .iter()
        if !already_taken {
            return Some(letter);  // BUG: no correlation to instance_id
        }
    }
    None
}
```

**The `_instance_id` parameter is prefixed with underscore â€” it is NEVER USED.**

The function simply returns the first fixed drive letter (A-Z) that:
1. Is a fixed drive (`GetDriveTypeW` returns `DRIVE_FIXED`)
2. Has an existing path (`Path::exists`)
3. Hasn't been assigned to a previously enumerated disk

This means:
- First enumerated disk â†’ gets 'C' (first fixed drive)
- Second enumerated disk â†’ gets 'D' (next fixed drive)
- etc.

The assignment is purely ordinal, with ZERO correlation to the actual disk instance_id.

**Why this broke:**
- The SanDisk USBSTOR plugged at F: was the first USB fixed disk enumerated.
- `find_drive_letter_for_instance_id` returned 'C' because C: is the first fixed drive.
- The audit log reports `drive_letter: "C"` for the SanDisk.

### Fix Required

Properly correlate the SetupDi instance_id to the drive letter using `IOCTL_STORAGE_GET_DEVICE_NUMBER` or `CM_Get_Child`/`CM_Get_Sibling` walk to match the disk device interface to the volume mount point.

The correct approach:
1. For each fixed drive letter, open `\\.\X:`
2. Send `IOCTL_STORAGE_GET_DEVICE_NUMBER` to get the device number
3. Match against the device number from the SetupDi enumeration
4. Alternatively, use `GetVolumePathNamesForVolumeNameW` + `FindFirstVolumeW` / `FindNextVolumeW` to enumerate volume GUIDs, then `GetVolumeNameForVolumeMountPointW` to map drive letters to volume GUIDs, then match volume GUIDs to disk instance IDs via `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS`.

Simpler approach: Use `SetupDiEnumDeviceInterfaces` with `GUID_DEVINTERFACE_DISK`, then for each interface, call `GetVolumePathNamesForVolumeNameW` after obtaining the device path. Or: enumerate volume GUIDs and use `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS` to find the PhysicalDriveN for each volume, then match PhysicalDriveN to the SetupDi instance_id.

**Recommended fix:** Use `FindFirstVolume` / `FindNextVolume` to enumerate all volumes, then for each volume:
1. Get mount points (drive letters) via `GetVolumePathNamesForVolumeMountPointW`
2. Get disk extents via `IOCTL_VOLUME_GET_VOLUME_DISK_EXTENTS` on the volume handle
3. Match the disk number to the SetupDi `instance_id` via `IOCTL_STORAGE_GET_DEVICE_NUMBER` on `\\.\PhysicalDriveN`

This is a non-trivial change to `dlp-common/src/disk.rs`.

## Recommended Code Changes

### Change 1: Fix `find_drive_letter_for_instance_id` (disk.rs)

**File:** `dlp-common/src/disk.rs`
**Lines:** 772-809
**Action:** Rewrite to properly correlate instance_id to drive letter using volume-to-disk mapping.

### Change 2: Add DACL deny-all method (device_controller.rs)

**File:** `dlp-agent/src/device_controller.rs`
**Action:** Add `set_volume_deny_all(drive_letter)` method, modeled after `set_volume_readonly` but with a deny-all SDDL string.

SDDL for deny-all (except SYSTEM):
```
D:(D;;FA;;;S-1-1-0)(D;;FA;;;S-1-5-11)(A;;FA;;;S-1-5-18)
```

This denies Full Access to Everyone and Authenticated Users, while allowing SYSTEM (for system operations).

### Change 3: Fix WR-01 race (usb.rs)

**File:** `dlp-agent/src/detection/usb.rs`
**Lines:** 460-484 (`on_usb_device_arrival`)
**Action:** After parking identity in `pending_identity`, also check if there's an un-identity'd drive in `blocked_drives`. If exactly one exists, assign the identity and call `apply_tier_enforcement`.

### Change 4: Fix startup gap (usb.rs / service.rs)

**File:** `dlp-agent/src/detection/usb.rs` and `dlp-agent/src/service.rs`
**Action:** After `scan_existing_drives`, attempt to capture identities for already-plugged USB devices. Use `SetupDi` enumeration to find USB devices with drive letters, populate `device_identities`, and call `apply_tier_enforcement` for each.

Alternatively, add a `scan_existing_usb_identities` method to `UsbDetector` that enumerates already-present USB mass storage devices and populates both `blocked_drives` and `device_identities` with correct identity data, then calls `apply_tier_enforcement`.

### Change 5: Apply both enforcement layers for Blocked tier

**File:** `dlp-agent/src/detection/usb.rs`
**Lines:** 406-426 (`apply_tier_enforcement` Blocked branch)
**Action:** In addition to `disable_usb_device`, also call `set_volume_deny_all` for Blocked tier. This provides defense-in-depth: if PnP disable fails or is bypassed, the DACL still blocks I/O.

```rust
UsbTrustTier::Blocked => {
    // Layer 1: PnP disable
    if let Err(e) = controller.disable_usb_device(...) { ... }
    // Layer 2: DACL deny-all (defense-in-depth)
    if let Err(e) = controller.set_volume_deny_all(letter) { ... }
}
```
