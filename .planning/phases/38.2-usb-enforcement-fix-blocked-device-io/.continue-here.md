## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Notify-based watcher mistaken for enforcement | The `notify` crate is a passive file system watcher (observe AFTER write). Treating its events as an enforcement point produces audit-only DENY decisions while writes succeed. This is the live bug being fixed. | advisory | The locked decision (Area 1) treats `notify` as observability ONLY. PnP-disable + DACL deny-all are the real-time enforcement layers. |
| User-mode API hooking for DLP enforcement | DLL injection / IAT hooking conflicts with EDR products (gets quarantined as malware), can't inject into PPL processes, has WoW64 split, is trivially bypassed via cmd.exe/PowerShell, and doesn't catch SYSTEM/kernel I/O. NOT real defense in depth. | blocking | The locked decision (Area 1) explicitly rejects API hooking. Plan MUST use PnP + DACL only. |

## 3 Remaining Gray Areas (resume here)

### Area 2: AGENT-UNKNOWN agent_id

**Framing:** `service.rs:759` reads `DLP_AGENT_ID` env var, defaults to `"AGENT-UNKNOWN"`. Per `docs/OPERATIONAL.md`, admins are expected to set this via `setx /M DLP_AGENT_ID "AGENT-XXX-001"` before installing the service. The "AGENT-UNKNOWN" in the live audit log just means the user hasn't done that step.

**Single question to ask:**
- Is this in scope for 38.2 (e.g., add a startup check that warns if `DLP_AGENT_ID` is unset, similar to the Phase 31 UI binary warning)?
- Or out of scope (existing operational doc covers it; admin error, not a code bug)?
- Or split to a separate phase (e.g., 38.3 — operational hardening: warn on missing config, validate DLP_AGENT_ID format, persist to registry)?

### Area 3: Re-plug & tier-change semantics

**Framing:** `CM_Disable_DevNode` with `CM_DISABLE_ABSOLUTE` persists across re-plugs of the same VID/PID/serial — the device stays disabled until `enable_usb_device` runs. Currently `enable_usb_device` is only called from `on_usb_device_removal` (line ~621 of detection/usb.rs per Phase 31 verification). If admin changes a device's tier from Blocked → ReadOnly/FullAccess in the registry, today nothing wires `enable_usb_device`.

**Questions to ask:**
1. When admin changes tier Blocked → ReadOnly/FullAccess via the registry, where should the re-enable fire? (a) Registry-cache poll detects the tier change and calls enable_usb_device immediately. (b) Wait for next user re-plug. (c) Manual admin action.
2. For DACL tier change ReadOnly → FullAccess: should `restore_volume_acl()` fire automatically on tier-change, or only on device removal as today?

### Area 4: Drive-letter mislabel ("C" for SanDisk)

**Framing:** The audit log shows `DISK_DISCOVERY` event with drive_letter "C" for the SanDisk USBSTOR — but the user reports plugged at F:. That code path is in `dlp-agent/src/detection/disk.rs` (Phase 33 disk-enum), NOT the Phase 31 USB watcher. Phase 31's WR-01 anti-pattern flagged a similar issue (race between USB_DEVICE and VOLUME notifications) but for a different code path.

**Single question to ask:**
- In scope for 38.2 (it's a USB-related bug, surfaced together)?
- Out of scope — split to a Phase 33 follow-up (it's a disk-enum bug, separate concern)?
- Just note it as a deferred idea for the roadmap?
