# Resume Plan — Phase 38.2 USB Enforcement Fix

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Notify-based watcher mistaken for enforcement | The `notify` crate is a passive file system watcher (observe AFTER write). Treating its events as an enforcement point produces audit-only DENY decisions while writes succeed. This is the live bug being fixed. | advisory | Plans must enforce at PnP level (`CM_Disable_DevNode`) or via a real I/O filter (minifilter / API hook) — never rely on `notify` to block. |
| `UsbEnforcer::check` returns `None` for registered devices | Phase 26 D-07 design says registered tiers (Blocked/ReadOnly/FullAccess) all return `None` because `DeviceController` is supposed to enforce at PnP. If `DeviceController` doesn't fire, registered Blocked devices have NO enforcement. | advisory | Verify the `usb_wndproc` arrival handler actually dispatches `DeviceController::disable_usb_device` for Blocked-tier registered devices BEFORE relying on the `None` return. |
