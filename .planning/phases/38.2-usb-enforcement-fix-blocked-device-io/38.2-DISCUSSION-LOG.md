### Q4. Which I/O interception layer for 38.2?

| Option | Description | Selected |
|--------|-------------|----------|
| Volume DACL deny-all | NTFS-level deny-all on volume root for Blocked tier | |
| Minifilter driver | Kernel-mode FltMgr filter; full I/O interception | |

| Step back — PnP + DACL only | Stay with the two OS-enforced layers | |

**User's choice:** API hooking (initially) — then redirected to PnP + DACL after pitfalls were explained.
**Notes:** Initial selection was API hooking. Explained concrete blockers: EDR conflict (Defender/CrowdStrike/SentinelOne quarantine third-party hooking as malware), PPL processes inaccessible, WoW64 split, AppInit_DLLs deprecated, trivially bypassed via cmd.exe / PowerShell, doesn't catch SYSTEM/kernel I/O, CET shadow-stack incompatibility on Win11. User reconsidered.

---

### Q5. After hearing API hooking pitfalls, do you still want it?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep API hooking | Accept the EDR / bypass risks | |
| Switch to PnP + DACL | Drop API hooking; rely on the two OS-enforced layers | |
| Switch to PnP + DACL, defer minifilter | Same as above; explicitly defer minifilter to v0.8.0+ | |

**User's choice:** Asked clarifying question: "do PnP + DACL provide real-time defending?"
**Notes:** User wanted to verify that dropping API hooking did not weaken the real-time-defense property. Answered: yes — PnP fires at the device-tree layer (lowest), DACL fires at the NTFS check (before write reaches disk). Both are OS-enforced and synchronous; neither relies on the agent observing a write after the fact. (Contrast with `notify`-based watcher, which is purely observational.)

**Notes:** Final lock for Area 1. Confirmed: scope = PnP + Volume DACL deny-all. API hooking REJECTED with rationale. Minifilter DEFERRED to v0.8.0+. `notify`-based watcher demoted to observability ONLY. See CONTEXT.md D-01..D-06.

## Area 3: Re-plug & tier-change semantics

### Q1. PnP re-enable on Blocked → ReadOnly/FullAccess tier change

| Option | Description | Selected |
|--------|-------------|----------|
| Registry-cache poll detects | 30s poll loop detects tier change; calls `enable_usb_device` immediately | |
| Wait for re-plug (status quo) | `enable_usb_device` only fires on physical removal; admin instructs user to re-plug | ✓ |

## Deferred Ideas

- **Windows minifilter driver (FltMgr)** — kernel-level I/O interception on top of PnP + DACL; only path to a per-process exception model. Multi-week scope. Target v0.8.0+.
- **Per-process exception lists** — requires minifilter to be useful (DACL is process-agnostic). Out of 38.2.
- **`AGENT-UNKNOWN` operational hardening** — Phase 38.3.
- **Tier-change reactivity (poll-driven re-enable)** — considered and rejected for 38.2 due to race-window complexity. May be revisited if admin UX feedback shows physical-re-plug friction is too high.
