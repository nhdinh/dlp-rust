<objective>
Close the APP-05 agent-side gap: `dlp-agent/src/ipc/pipe3.rs` currently uses `..` in the
`ClipboardAlert` destructure, silently discarding `source_application` and
`destination_application`. `AuditEvent` already has `with_source_application()` and
`with_destination_application()` builder methods (dlp-common/src/audit.rs). This plan
extracts the two identity fields from the destructure and chains them onto the `AuditEvent`.

After this plan APP-05 is fully satisfied: clipboard audit events in `audit.jsonl` will carry
populated `source_application` / `destination_application` whenever the UI resolves identity.
</objective>

<action>
In `dlp-agent/src/ipc/pipe3.rs`, locate the `Pipe3UiMsg::ClipboardAlert` destructure pattern
(around line 193). Change it from:

```rust
Pipe3UiMsg::ClipboardAlert {
    session_id,
    classification,
    preview,
    text_length,
    ..
} => {
```

to:

```rust
Pipe3UiMsg::ClipboardAlert {
    session_id,
    classification,
    preview,
    text_length,
    source_application,
    destination_application,
    ..
} => {
```

Then find the `AuditEvent` construction block (after the tier match). The current code ends with:

```rust
event = event.with_access_context(dlp_common::AuditAccessContext::Local);
crate::audit_emitter::emit(&event).ok();
```

Add identity builder calls between those two lines:

```rust
event = event.with_access_context(dlp_common::AuditAccessContext::Local);
event = event.with_source_application(source_application);
event = event.with_destination_application(destination_application);
crate::audit_emitter::emit(&event).ok();
```

No new imports are needed: `with_source_application` / `with_destination_application` are methods
on `dlp_common::AuditEvent` (already in scope via `dlp_common::`), and `AppIdentity` is the
concrete type carried in `Pipe3UiMsg::ClipboardAlert` from dlp-common (already a transitive dep).
</action>
