<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| classify_and_alert → send_clipboard_alert | AppIdentity structs passed by value — no pointer aliasing |
| send_clipboard_alert → Pipe 3 | JSON-serialized AppIdentity transmitted to dlp-agent — agent treats these as untrusted input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Spoofing | Binary rename bypass | mitigate | Inherited from Plan 01. Path-keyed cache; WinVerifyTrust verifies file bytes, not name. |
| T-25-02 | Denial of Service | WinVerifyTrust network block | mitigate | Inherited from Plan 01. `WTD_REVOCATION_CHECK_NONE` eliminates network calls. |
| T-25-06 | Tampering | Agent receives forged AppIdentity via Pipe 3 | accept | Pipe 3 runs on a local named pipe — only the UI process (same user session) can write to it. An attacker in the same session already has equivalent privilege. Agent validates JSON structure but cannot independently re-verify the Authenticode claim (it runs in session 0 without access to user-session files). Out of scope for Phase 25; accepted as architecture constraint documented in RESEARCH.md Architectural Responsibility Map. |
| T-25-07 | Information Disclosure | publisher string leaks process identity via Pipe 3 JSON | accept | `publisher` and `image_path` are captured to enforce DLP policy — intentional data capture. Pipe 3 JSON is accessible only to the SYSTEM-level agent. No external network transmission in this phase. |
</threat_model>
