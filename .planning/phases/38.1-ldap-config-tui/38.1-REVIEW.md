## Warnings

### WR-01: Missing integration tests for cache_ttl_secs validation edge cases

**File:** `dlp-admin-cli/src/screens/dispatch.rs:1343-1369`
**Issue:** The `handle_ldap_config_editing` numeric validation branch tests the `[60, 3600]` range, but the existing `ldap_config_tests` module only covers static predicate functions (`ldap_is_bool`, `ldap_is_numeric`, constants, and key order). There are no tests verifying:

1. That a value of `59` is rejected with the correct error message
2. That a value of `3601` is rejected
3. That a value of `60` is accepted and stored as `serde_json::Value::Number`
4. That a value of `3600` is accepted
5. That non-numeric input (e.g., "abc") is rejected
6. That the form stays in edit mode on rejection (buffer preserved, `editing` stays `true`)
7. That the form exits edit mode on success (buffer cleared, `editing` set to `false`)

The Alert config equivalent (`handle_alert_config_editing`) has the same gap, but this is new code in this phase and should meet the project's testing standards (CLAUDE.md 9.8: "MUST write unit tests for all new functions and types").

**Fix:** Add integration-style unit tests in the `ldap_config_tests` module that construct an `App` with `Screen::LdapConfig`, simulate key events through `handle_event`, and assert on the resulting screen state and status messages. Example test skeleton:

```rust
#[test]
fn ldap_cache_ttl_rejects_below_min() {
    let mut app = make_test_app(Screen::LdapConfig {
        config: serde_json::json!({"cache_ttl_secs": 300}),
        selected: 3,
        editing: true,
        buffer: "59".to_string(),
    });
    handle_event(&mut app, AppEvent::Key(enter_key()));
    assert!(matches!(app.screen, Screen::LdapConfig { editing: true, .. }));
    let (msg, kind) = app.status.as_ref().expect("status set");
    assert!(msg.contains("between 60 and 3600"), "was: {msg}");
    assert_eq!(*kind, StatusKind::Error);
}
```

## Info

### IN-01: Pre-existing `nav()` helper has underflow/panic risk on empty menus

**File:** `dlp-admin-cli/src/screens/dispatch.rs:65-74`
**Issue:** The shared `nav()` helper computes `count - 1` without checking if `count == 0`. If any caller ever passes `count = 0`, this will panic on underflow (usize wrap in debug, panic in release on modern Rust). The LDAP code passes `LDAP_ROW_COUNT = 7`, so this is not triggered by the new code. However, this is a latent correctness issue in shared infrastructure.

**Fix:** Add a guard at the top of `nav()`:

```rust
fn nav(selected: &mut usize, count: usize, key: KeyCode) {
    if count == 0 {
        return;
    }
    // ... existing logic
}
```
