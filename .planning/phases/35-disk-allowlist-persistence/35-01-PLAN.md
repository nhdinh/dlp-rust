                        serial: None,
                        size_bytes: None,
                        is_boot_disk: false,
                        encryption_status: None,
                        encryption_method: None,

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem (`agent-config.toml`) -> AgentConfig | Untrusted TOML input crosses into in-process state. The file lives at `C:\ProgramData\DLP\agent-config.toml` which is ACL-protected (SYSTEM/admin write only), but the parsed `disk_allowlist` array of structs needs validation that an admin-edited or tampered file cannot crash or DoS the agent. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-35-01 | Tampering | `[[disk_allowlist]]` admin-editable TOML section | mitigate | File ACL on `C:\ProgramData\DLP\` already restricts write access to SYSTEM/admin (per Phase 33 hardening). `instance_id` from TOML is used only as a `HashMap<String, _>` key in `DiskEnumerator.instance_id_map` and as a TOML string -- never used in filesystem, OS, or shell calls. No injection surface in this plan (Plan 35-02 will introduce the use sites; this plan only adds the field shape and tests). |
| T-35-02 | DoS (availability) | Malformed `[[disk_allowlist]]` entry causing TOML parse failure | mitigate | Existing `AgentConfig::load()` (lines 200-219) catches parse errors and falls back to `Default` with a `warn!` log. `#[serde(default)]` on the new field plus existing struct-level `#[serde(default)]` on `DiskIdentity` mean a partial entry yields a default value, not a parse failure. The agent continues without the corrupted allowlist rather than crashing. |
| T-35-03 | Information Disclosure | Encryption status fields persisted in plaintext TOML | accept | Encryption metadata (status, method, last-checked timestamp) is non-sensitive operational telemetry; ASVS V6 does not apply. Per D-08 the fields are persisted verbatim; per D-09 they are not re-written after the Phase 34 re-check. The `#[serde(skip_serializing_if = "Option::is_none")]` annotations ensure absent fields stay absent (Test 3 proves this). |
