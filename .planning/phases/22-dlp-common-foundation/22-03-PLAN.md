<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dlp-user-ui process ↔ dlp-agent service (Pipe 3, length-prefixed JSON frames) | UI process runs in the interactive user session; agent service runs as SYSTEM. Pipe 3 is one-way (UI → agent). Messages on this pipe are an enforcement signal (UI-observed clipboard events) that the agent consumes. This plan extends the `ClipboardAlert` frame with new fields that flow from the less-privileged (user-session) side to the more-privileged (SYSTEM) side. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-11 | Spoofing | Malicious user-session process connects to Pipe 3 and sends forged `ClipboardAlert` frames with spoofed `source_application: Some(trusted_publisher)` to influence policy evaluation | accept (pre-existing risk) | This risk pre-dates Phase 22. Pipe 3 ACL already restricts the pipe to the registered UI process (established in the v0.2.0 IPC hardening). Adding new fields to a variant does not widen the attack surface — an attacker already able to write to Pipe 3 could forge any field. Mitigation lives in the pipe ACL layer, not in the message struct. |
| T-22-12 | Tampering | IPC struct divergence between the two duplicated `messages.rs` files (D-14 deliberate duplication + D-15 required identical extension) causes silent runtime protocol mismatch — one side populates fields, the other ignores them | mitigate | Task 2 acceptance criteria include a `diff` check on the new-field lines between the two files. Pitfall 3 in RESEARCH flags this exact failure mode; the byte-for-byte mirror requirement is how Plan 03 counters it. Future work (out of scope this phase) could consolidate into `dlp-common` to make divergence impossible. |
| T-22-13 | Information Disclosure | `ClipboardAlert.source_application.image_path` contains file-system paths that may reveal installed software inventory to whoever can read audit logs downstream | accept | The image_path is already audit-logged today via `AuditEvent.application_path` (existing, pre-Phase-22). Phase 22 does not expand the read surface — it just adds a typed form of the same information. Downstream audit storage access control (Phase 9 admin audit logging) is the enforcement tier. |
| T-22-14 | Denial of Service | Attacker sends a `ClipboardAlert` frame with an enormous `image_path` String to exhaust memory on the agent side during JSON parse | accept | The IPC framing layer already enforces a 4-byte-length-prefix cap (see `dlp-agent/src/ipc/` framing code — existing safeguard). Adding two new `Option<String>-nested-inside-struct` fields does not change the maximum frame size; the length-prefix cap still bounds allocation. |
| T-22-15 | Elevation of Privilege | Forging `destination_application: Some({publisher: "Trusted Corp"})` could make the agent evaluator allow a paste that should be blocked | accept | Same response as T-22-11: Pipe 3 ACL + UI registration (`RegisterSession` on Pipe 1) scopes who can send. Policy evaluation logic (Phase 26) must still treat application identity as subject input, not as verified identity; verification happens in `dlp-user-ui` via `WinVerifyTrust` in Phase 25. The untrusted nature of user-session IPC input is an architectural invariant unchanged by this plan. |

No HIGH-severity threats newly introduced by this plan. T-22-12 (the duplicated-file divergence) is a MEDIUM concern that is explicitly mitigated by the acceptance-criteria diff check in Task 2. All other threats pre-date Phase 22 and live in architectural layers outside this plan's scope.
