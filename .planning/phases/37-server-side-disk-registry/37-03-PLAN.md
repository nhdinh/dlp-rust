<objective>
Close the D-02 / D-03 loop on the agent side: extend `AgentConfigPayload` (the agent's mirror of the server's wire-format) with the `disk_allowlist` field, then teach `config_poll_loop` to detect and apply server-pushed allowlist changes -- merging them into `DiskEnumerator.instance_id_map` without clobbering live-enumerated disks (Pitfall 5).

Purpose: Without this plan, an admin's add/remove on the server side reaches the SQLite registry and the wire payload, but the agent never updates its in-memory enforcement map. Phase 36's enforcement still uses the old map until the next service restart. This plan makes "next poll cycle" the actual update boundary.

Output: Agent-side AgentConfigPayload mirror has the `disk_allowlist` field; config_poll_loop diffs it against cfg.disk_allowlist; on change it (a) records 'disk_allowlist' in changed_fields, (b) merges entries into instance_id_map (add new, remove deleted, preserve unrelated live entries), (c) updates cfg.disk_allowlist, (d) the existing cfg.save call persists the new TOML.
</objective>

                    if !changed_fields.is_empty() {

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dlp-server JSON response → agent config_poll_loop | The agent treats `disk_allowlist` from `GET /agent-config/{id}` as authoritative. The endpoint is HTTP (often plaintext). The trust gradient is: TLS or local network only -- if MITM is feasible an attacker could inject allowlist entries. This phase does not add TLS; the existing transport posture applies. |
| agent in-memory state → DiskEnumerator.instance_id_map | The merge must not lose live-enumerated disks (Pitfall 5). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-10 | Tampering | malicious /agent-config/{id} response injecting an attacker-controlled `disk_allowlist` to allow exfiltration | accept | Existing transport posture (no TLS pinning) is unchanged by this phase; mitigation belongs to a separate Zero-Trust transport phase. The attacker would already need to MITM HTTP traffic. Documented in PROJECT.md as accepted residual risk. |
| T-37-11 | Tampering | local merge clobbers live-enumerated disks not in server allowlist (Pitfall 5) | mitigate | Merge logic explicitly preserves entries in `instance_id_map` whose `instance_id` is NOT in the OLD allowlist. Only entries that were in the OLD allowlist but are absent from the NEW are removed. Live-enumerated entries are tracked separately by Phase 33's enumeration task and never appear in `cfg.disk_allowlist`. |
| T-37-12 | Denial of Service | repeated polling triggers spurious diff and TOML rewrite | mitigate | Diff comparison is `cfg.disk_allowlist != payload.disk_allowlist` (PartialEq derives on DiskIdentity, all fields). When unchanged, the diff returns false and no work occurs (no log line, no TOML write). Tested by Test 7. |
| T-37-13 | Race / Lock-order inversion | config mutex + instance_id_map write lock acquired in different orders by config_poll_loop and Phase 36 enforcement | mitigate | This plan ALWAYS acquires the config mutex FIRST and releases it BEFORE acquiring instance_id_map.write(). Phase 36 enforcement only takes instance_id_map.read() -- documented invariant: "config -> map" is the only allowed order. Code must drop the cfg lock guard before calling get_disk_enumerator. |
</threat_model>
