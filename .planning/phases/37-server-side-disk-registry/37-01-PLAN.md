<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → admin REST API | HTTP request body crosses untrusted-to-trusted at /admin/disk-registry endpoints (handled in Plan 02; Plan 01 only ships the validation surface inside the DB CHECK constraint). |
| admin REST API → SQLite | Validated, authenticated payloads cross to the persistent store. CHECK constraint is the last line of defense. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-01 | Tampering | disk_registry.encryption_status | mitigate | DB CHECK(encryption_status IN ('fully_encrypted','partially_encrypted','unencrypted','unknown')) — schema-level allowlist defended in this plan; Plan 02 adds the handler-level allowlist for early 422 rejection. |
| T-37-02 | Tampering | (agent_id, instance_id) duplicate insert ("pre-register attack") | mitigate | UNIQUE(agent_id, instance_id) constraint on disk_registry; pure INSERT in DiskRegistryRepository::insert lets the conflict surface as rusqlite::Error (extended_code 2067), which Plan 02 maps to 409 Conflict. |
| T-37-03 | Tampering | DiskRegistryRepository::upsert (does NOT exist) | mitigate | This plan deliberately does not implement an `upsert` method — only `insert`. Pure INSERT semantics (D-05) prevent silent overwrites that could downgrade an `encryption_status`. |
</threat_model>
