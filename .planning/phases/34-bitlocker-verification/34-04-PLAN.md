<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `agent-config.toml` (filesystem) → service startup | Admin-controlled `[encryption].recheck_interval_secs` flows from disk into the spawned task's loop period. Already clamped + warn-logged in `AgentConfig::resolved_recheck_interval` (Plan 34-02). This plan only forwards the resulting `Duration`. |
| Service startup ordering | `set_encryption_checker` MUST precede `spawn_encryption_check_task` so the task can find its singleton. Wrong order = silent panic-or-noop in the spawned task. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-06 | Tampering / Configuration injection | `recheck_interval` value | mitigate (defense-in-depth) | Already mitigated upstream by Plan 34-02's clamp `[300, 86400]` + `warn!` on out-of-range. This plan adds no new path that bypasses the clamp — `agent_config.resolved_recheck_interval()` is the single source. The `info!` log line at the spawn site captures the *applied* (post-clamp) value, so an admin can audit what the agent actually used. |
| T-34-07 | Denial of Service / Startup deadlock | Encryption task blocks on disk enumeration | accept | Per D-04, `spawn_encryption_check_task` internally polls `DiskEnumerator::is_ready` with 250ms sleep — the spawn returns immediately and the wait happens on the spawned task's runtime thread. No await on disk enumeration completion in `service.rs`. |
| T-34-08 | Repudiation | Misordered singleton registration silently breaks verification | mitigate | The order `set_encryption_checker` → `spawn_encryption_check_task` mirrors the proven Phase-33 pattern (lines 626-627). The acceptance criteria require this ordering literally. |
</threat_model>
